# UX Research рекомендации — AI Portal

Дата: 2026-04-03
Версия: 1.0

---

## 1. Введение

Данный документ содержит анализ текущего состояния UI/UX AI Portal (на этапе проектирования), рекомендации по улучшению usability и идеи для A/B тестирования ключевых взаимодействий, особенно связанных с AI-генерацией контента.

**Важно:** Поскольку продукт находится на стадии проектирования, анализ текущего UI основан на лучших практиках аналогичных продуктов (Notion, Confluence, GitBook) и гипотетических решениях для AI Portal. После создания первых прототипов данный документ должен быть обновлён с реальными скриншотами и user testing данными.

---

## 2. Анализ текущего UI (на основе референсов и лучших практик)

### 2.1. Что должно быть хорошо (сильные стороны подхода)

#### 2.1.1. AI-first подход снижает когнитивную нагрузку
**Проблема традиционных вики:** Пользователь должен знать:
- Где создать страницу
- Какую структуру выбрать
- Как форматировать контент
- Какие шаблоны использовать

**Решение AI Portal:** Пользователь просто описывает, что ему нужно, на естественном языке. AI делает остальное.

**UX-преимущество:**
- Нулевой порог входа для создания контента
- Не нужно обучать структуре и форматированию
- Быстрый time-to-value (страница за секунды)

#### 2.1.2. Единый интерфейс вместо множества инструментов
В отличие от комбинации Notion + GitBook + Confluence, AI Portal предлагает единое решение. Пользователь не переключается между инструментами.

#### 2.1.3. Self-hosted с простым развёртыванием
Docker Compose = одна команда для запуска. Это критично для IT-администраторов, которые не хотят тратить дни на настройку.

---

### 2.2. Потенциальные проблемы UX (риски)

#### 2.2.1. «Пустой экран» — что делать после входа?
**Риск:** Пользователь входит в систему и видит пустой портал. Нет подсказок, что делать дальше.

**Аналогия:** Notion решает это через templates gallery и onboarding tour. Confluence — через space blueprints.

**Рекомендация:**
- Onboarding tour при первом входе
- Quick start guide: «Создайте первое пространство» → «Сгенерируйте страницу промптом»
- Примеры промптов для разных сценариев

#### 2.2.2. AI-генерация: непредсказуемость результата
**Риск:** Пользователь вводит промпт, AI генерирует контент, но результат не соответствует ожиданиям. Пользователь не понимает, как улучшить промпт.

**Аналогия:** ChatGPT решает это через suggestions и возможность перегенерации.

**Рекомендация:**
- Примеры хороших промптов рядом с input полем
- Кнопка «Улучшить промпт» (AI помогает переформулировать)
- Быстрая перегенерация с минимальным friction
- Индикатор качества/уверенности AI (опционально)

#### 2.2.3. Навигация в большом портале
**Риск:** Когда портал растёт до 100+ страниц, навигация становится сложной. Дерево страниц в sidebar может стать нечитаемым.

**Аналогия:** Notion решает это через поиск (Cmd+K) и recently viewed. Confluence — через space hierarchy и labels.

**Рекомендация:**
- Глобальный поиск всегда доступен (Cmd+K / Ctrl+K)
- Recently viewed pages в sidebar или dropdown
- breadcrumbs на каждой странице
- Возможность «закрепить» важные пространства/страницы

#### 2.2.4. Редактирование AI-контента: где граница?
**Риск:** Пользователь не понимает, что можно редактировать сгенерированный контент, или боится «сломать» AI-результат.

**Рекомендация:**
- Визуальный индикатор: «Этот контент сгенерирован AI» (badge)
- Ясное сообщение: «Вы можете редактировать этот контент как обычную страницу»
- Кнопка «Перегенерировать» рядом с кнопкой «Редактировать»

#### 2.2.5. Права доступа: сложность для нетехнических пользователей
**Риск:** Менеджеры и HR не понимают разницу между ролями Admin/Editor/Viewer и настройками public/private пространств.

**Аналогия:** Notion решает это через простой UI: «Share» → «Invite people» → выбор роли.

**Рекомендация:**
- Простой UI для шаринга: «Поделиться пространством» → выбор людей и ролей
- Объяснение ролей простыми словами:
  - **Администратор:** Может управлять всем, включая настройки портала
  - **Редактор:** Может создавать и редактировать страницы
  - **Читатель:** Может только просматривать страницы
- Visual feedback: иконка замка для private пространств

## 3. Рекомендации по usability

### 3.1. Onboarding (первый опыт пользователя)

#### 3.1.1. Первый вход
**Цель:** Пользователь должен создать первую страницу за < 2 минуты.

**Рекомендуемый flow:**
1. Регистрация/вход
2. Welcome screen: «Добро пожаловать в AI Portal!»
3. Краткое объяснение: «Создавайте страницы и пространства через AI-промпты»
4. CTA: «Создать первое пространство» или «Посмотреть примеры»
5. Интерактивный tour (опционально, можно пропустить):
   - «Вот sidebar с пространствами»
   - «Здесь вы вводите промпт для AI»
   - «Здесь отображается сгенерированная страница»
6. Предложение примера промпта: «Попробуйте: "Создай страницу онбординга для нового сотрудника"»

**Metrics to track:**
- Time to first page created
- % пользователей, завершивших onboarding
- % пользователей, пропустивших tour

---

#### 3.1.2. Примеры промптов
**Проблема:** Пользователи не знают, что писать в AI.

**Решение:** Gallery примеров промптов для разных сценариев:

| Сценарий | Пример промпта |
|---|---|
| Команда | «Создай страницу команды backend-разработки с описанием ролей, процессов code review и meeting cadence» |
| API документация | «Создай документацию для API endpoint POST /users с параметрами: name (string, required), email (string, required), role (enum: admin, user)» |
| Онбординг | «Создай страницу онбординга для нового сотрудника: первый день, первая неделя, первый месяц» |
| FAQ | «Создай FAQ по отпускам и больничным: как оформить, кому обращаться, сроки» |
| Meeting notes | «Создай шаблон для meeting notes: дата, участники, agenda, action items, decisions» |
| Release notes | «Создай страницу релиза версии 2.5: новые функции, исправления багов, breaking changes» |

**UX реализация:**
- Модальное окно «Примеры промптов» при первом использовании AI
- Кнопка «Вставить пример» рядом с полем ввода
- Автодополнение промптов на основе контекста (если пользователь в пространстве «HR», предложить HR-шаблоны)

### 3.4. Редактор страниц

#### 3.4.1. Rich-text vs Markdown
**Проблема:** Разные пользователи предпочитают разные форматы.

**Рекомендация:** Поддержка обоих режимов с переключением:
- **Rich-text mode:** WYSIWYG, как в Notion/Confluence (для нетехнических пользователей)
- **Markdown mode:** Raw Markdown editing (для разработчиков)
- Переключатель в toolbar: «Rich Text | Markdown»
- Синхронизация между режимами (изменения в одном отражаются в другом)

#### 3.4.2. Toolbar рекомендации
**Базовый toolbar:**
```
[H1] [H2] [H3] | [Bold] [Italic] [Code] | [Link] [Image] | [List] [Numbered] | [Quote] [Divider] | [AI ✨]
```

**Advanced toolbar (expandable):**
```
[Table] [Code block] [Embed] [Mermaid] [Callout] [Toggle]
```

---

### 3.5. Обратная связь и ошибки

#### 3.5.1. Toast notifications
**Рекомендуемые типы:**
- ✅ **Success:** «Страница сохранена»
- ❌ **Error:** «Ошибка сохранения. Попробуйте снова.»
- ⚠️ **Warning:** «Вы покидаете страницу с несохранёнными изменениями»
- ℹ️ **Info:** «AI генерирует страницу...»

**UX правила:**
- Auto-dismiss через 3-5 секунд для success/info
- Persistent для error (требует действия пользователя)
- Не блокировать UI

#### 3.5.2. Обработка ошибок AI
**Сценарии:**
1. **API error (rate limit, invalid key):**
   - Сообщение: «Ошибка AI-провайдера. Проверьте настройки API ключа.»
   - CTA: «Перейти к настройкам»
2. **Timeout (генерация > 30 сек):**
   - Сообщение: «Генерация занимает больше времени. Попробуйте упростить промпт.»
   - CTA: «Попробовать снова»
3. **Empty response (AI вернул пустой результат):**
   - Сообщение: «AI не смог сгенерировать контент. Попробуйте более конкретный промпт.»
   - CTA: «Редактировать промпт»

## 4. A/B test идеи для AI-генерации

### 4.1. Test 1: Результат сразу в превью vs Draft mode

**Гипотеза:** Пользователи будут более удовлетворены, если увидят результат AI-генерации сразу в полноценном превью страницы (как final result), а не в отдельном draft-режиме.

**Вариант A (Draft mode):**
- AI генерирует контент
- Показывается в «draft» состоянии с явной рамкой/фоном
- Пользователь должен нажать «Сохранить», чтобы опубликовать
- Кнопки: «Редактировать», «Перегенерировать», «Сохранить», «Отменить»

**Вариант B (Immediate preview):**
- AI генерирует контент
- Показывается как обычная страница (без явного draft состояния)
- Auto-save через 2 секунды после генерации
- Пользователь может сразу редактировать
- Undo доступен через history

**Metrics to track:**
- Time to publish (время от генерации до сохранения)
- % страниц, сохранённых без редактирования
- % страниц, где пользователь нажал «Перегенерировать»
- User satisfaction (survey после генерации)
- Error rate (случаи, когда пользователь потерял контент)

**Sample size:** Минимум 100 генераций на вариант
**Duration:** 2 недели

---

### 4.2. Test 2: Inline prompt input vs Modal prompt input

**Гипотеза:** Inline поле ввода промпта прямо на странице (в месте курсора) увеличит частоту использования AI-генерации по сравнению с модальным окном.

**Вариант A (Modal):**
- Пользователь нажимает кнопку «AI» или shortcut
- Открывается модальное окно с полем ввода
- После генрации модальное закрывается, результат вставляется

**Вариант B (Inline):**
- Пользователь нажимает кнопку «AI» или набирает `/ai`
- Появляется inline поле ввода прямо на странице (как slash commands в Notion)
- После генрации результат вставляется в место курсора

**Metrics to track:**
- Frequency of AI usage (кол-во генераций на пользователя в неделю)
- Time to complete prompt (время от открытия input до отправки)
- Abandonment rate (пользователь открыл input, но не отправил промпт)
- User preference (survey)

**Sample size:** Минимум 200 пользователей на вариант
**Duration:** 3 недели

---

### 4.3. Test 3: Примеры промптов видны всегда vs Только при первом использовании

**Гипотеза:** Постоянное отображение примеров промптов рядом с полем ввода увеличит качество промптов и удовлетворённость результатом.

**Вариант A (Always visible):**
- Примеры промптов отображаются в sidebar или под полем ввода постоянно
- Пользователь может кликнуть и вставить пример

**Вариант B (First-time only):**
- Примеры показываются только при первом использовании AI
- После этого поле ввода «чистое», примеры скрыты за кнопкой «Показать примеры»

**Metrics to track:**
- Prompt quality (оценка AI: длина, конкретность, наличие контекста)
- Regeneration rate (сколько раз пользователь перегенерирует)
- Time to first successful generation
- User satisfaction (survey)

**Sample size:** Минимум 150 пользователей на вариант
**Duration:** 2 недели

---

### 4.4. Test 4: AI badge visible vs AI badge subtle

**Гипотеза:** Явный badge «Сгенерировано AI» на страницах повысит прозрачность, но может создать предвзятость против AI-контента.

**Вариант A (Visible badge):**
- Яркий badge вверху страницы: «✨ Сгенерировано AI»
- Цвет: синий или фиолетовый (neutral/positive)
- Всегда виден, даже после редактирования

**Вариант B (Subtle badge):**
- Badge только при первом просмотре после генерации
- После первого редактирования badge исчезает
- Или badge в footer страницы, менее заметный

**Вариант C (No badge):**
- Нет явного индикатора AI-генерации
- Пользователь сам решает, упоминать ли это

**Metrics to track:**
- Trust in content (survey: «Насколько вы доверяете этому контенту?»)
- Editing behavior (сколько пользователей редактируют AI-контент)
- Perception of quality (survey)
- Adoption rate (частота использования AI)

**Sample size:** Минимум 200 пользователей на вариант
**Duration:** 3 недели

---

### 4.5. Test 5: Single-shot generation vs Iterative refinement

**Гипотеза:** Возможность итеративно уточнять промпт и видеть результат в реальном времени (как чат) повысит удовлетворённость по сравнению с single-shot генерацией (один промпт → один результат).

**Вариант A (Single-shot):**
- Пользователь вводит промпт
- AI генерирует результат
- Пользователь может перегенерировать с новым промптом (полная замена)

**Вариант B (Iterative chat):**
- Пользователь вводит промпт
- AI генерирует результат
- Пользователь может «уточнить»: «Сделай короче», «Добавь раздел про X»
- AI обновляет результат инкрементально
- История уточнений видна как чат

**Metrics to track:**
- Time to satisfactory result
- Number of iterations per generation
- User satisfaction (survey)
- Preference (который вариант предпочитают)

**Sample size:** Минимально 100 генераций на вариант
**Duration:** 3 недели

## 5. Рекомендации по user testing

### 5.1. Методология

**Тип теста:** Moderated usability testing + Unmoderated remote testing

**Участники:**
- 8-12 пользователей для moderated testing (качественные инсайты)
- 50-100 пользователей для unmoderated testing (количественные метрики)

**Сегменты участников:**
- 40% технические пользователи (разработчики, DevOps)
- 30% нетехнические пользователи (менеджеры, HR, маркетинг)
- 30%混合 (технические менеджеры, product owners)

---

### 5.2. Tasks для testing

**Task 1: Создание пространства**
> «Представьте, что вы руководитель команды разработки. Создайте пространство для вашей команды.»

**Success criteria:**
- Пользователь создаёт пространство за < 1 минуты
- Пользователь понимает, где ввести промпт для AI-генерации

**Task 2: Генерация страницы через AI**
> «Создайте страницу с документацией по API авторизации.»

**Success criteria:**
- Пользователь находит AI prompt input
- Пользователь формулирует промпт
- Пользователь доволен результатом (или понимает, как улучшить)

**Task 3: Редактирование сгенерированной страницы**
> «Отредактируйте сгенерированную страницу: добавьте раздел про rate limiting.»

**Success criteria:**
- Пользователь понимает, что можно редактировать
- Пользователь успешно добавляет контент

**Task 4: Поиск информации**
> «Найдите документацию по endpoint /users.»

**Success criteria:**
- Пользователь находит поиск
- Пользователь находит нужную страницу за < 10 секунд

**Task 5: Управление правами**
> «Поделитесь пространством с коллегой и дайте ему права редактора.»

**Success criteria:**
- Пользователь находит настройки доступа
- Пользователь понимает разницу между ролями

### 5.3. Metrics to collect

**Quantitative:**
- Task completion rate (%)
- Time on task (seconds)
- Error rate (количество ошибок/попыток)
- SUS score (System Usability Scale)
- NPS (Net Promoter Score)

**Qualitative:**
- Think-aloud протокол (что пользователь говорит вслух)
- Pain points (где пользователь застрял)
- Confusion points (где пользователь не понял, что делать)
- Delight points (что пользователю понравилось)

### 6.2. Key takeaways

1. **AI-genration — это и сила, и риск.** Пользователи будут в восторге от скорости, но разочарованы, если результат не соответствует ожиданиям. Инвестируйте в UX промптов и перегенерации.

2. **Onboarding критичен.** Первый опыт определит, останется ли пользователь. Сделайте так, чтобы первый промпт → первая страница занимали < 2 минуты.

3. **Навигация масштабируется.** Продумайте поиск, breadcrumbs и недавно просмотренные страницы с самого начала. Когда портал вырастет до 100+ страниц, будет поздно.

4. **A/B тестируйте AI-взаимодействия.** Нет «правильного» способа показать AI-результат. Тестируйте draft mode vs immediate preview, inline vs modal, и т.д.

5. **Прозрачность AI.** Пользователи должны понимать, что контент сгенерирован AI, и что они могут его редактировать. Badge и ясные сообщения помогут.

6. **User testing рано и часто.** Не ждите релиза. Тестируйте прототипы с 5-8 пользователями на каждой итерации.
